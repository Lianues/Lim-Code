# 分布式上下文边界

当 Skill 的输出会被当前回复之外的消费者读取，或会产生多份独立产物时，读取本文件。

## 核心原则

聊天记录不是共享记忆。只有写入文件、报告、状态表，或在 prompt 中内联的内容，才能被无共享上下文的消费者稳定继承。

## 结构属性触发

触发任一条件时，考虑外部化上下文：

- 当前 prompt 外存在必要上下文。
- 输出会被当前回复之外的人、agent、工具、CI、reviewer 或未来会话读取。
- 任务会产生多份独立产物、报告、审查结论或状态更新。
- 后续消费者需要验证、修订或接续当前产物。

小边界可以内联摘要；大边界应使用共享信息库。

## 按主题分类，不按时间顺序堆叠

不要把每轮聊天或每个 session 原样追加成流水账。共享信息应按主题组织，便于不同消费者只读相关内容。

可选主题示例：

- 索引：每个文件用途。
- 长期上下文：目标、术语、硬约束、边界。
- 状态：当前阶段、已完成项、阻断项、下一步。
- 决策：已采纳/拒绝的决策和理由。
- 证据：命令输出、URL、文件路径、测试结果。
- 报告：调研、实现、审查、复核结果。
- 交接：当前状态和下一步，不替代主题文件。

这些是示例，不是固定模板。只创建会被实际读取和维护的主题。

## 主 agent 回写规则

- 派发前：写入共享背景、任务切分、前置结论、阻断项和验收标准。
- 回收后：把每个消费者返回的结论、证据、阻断项和未验证点写成报告或更新状态。
- 修订后：记录“前置结论 -> 阻断项 -> 修正内容 -> 复核结果”的对应关系。
- 复核前：prompt 指向前置报告和本轮修订摘要，或直接列出完整内容。
- 收尾时：更新状态或交接摘要，让下一位消费者不依赖当前聊天记忆。

## 相对指代判定

判断标准：脱离当前聊天记录，读者能否唯一确定所指对象？如果不能，就是不合格指代。

不合格例子：

- “如上”
- “继续刚才”
- “只审查前面的问题”
- “参考另一个执行者的结论”

必须改为：

- 文件路径：前置报告、状态文件、决策文件。
- 内联摘要：前置结论、阻断项、修订摘要、被审对象、通过标准。
