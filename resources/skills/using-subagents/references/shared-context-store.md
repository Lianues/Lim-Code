# 共享信息库

当多个 subagents 需要共享事实、状态、上一轮结论或项目决策时，读取本文件。

## 核心原则

Subagent 无状态，所以主 agent 必须维护一个可读取的信息库。聊天记录不是信息库；只有写入文件、报告或状态表，并在 prompt 中明确要求读取的内容，才能被后续 subagent 稳定继承。

## 推荐结构

可参考 `docs/pm/` 模式，按任务规模选择轻重：

- `README.md`：信息库索引，说明每个文件用途。
- `MASTER_CONTEXT.md`：长期背景、目标、硬约束、术语和边界。
- `STATUS.json` 或 `STATUS.md`：当前阶段、已完成项、阻断项、下一步。
- `reports/`：每个 subagent 的调研、实现、审查、复核报告。
- `HANDOFF.md`：跨会话或长任务交接摘要。
- `DECISIONS.md`：主 agent 已采纳/拒绝的关键决策和理由。

小任务可以只用一个 `agent-context.md`；大任务应拆成索引、状态和报告。

## 主 agent 维护规则

1. 派发前：把共享背景、任务切分、上一轮阻断项和验收标准写入信息库，或在 prompt 中完整列出。
2. 回收后：把每个 subagent 的结论、证据、阻断项和未验证点写成报告或更新状态。
3. 修订后：记录“上一轮结论 -> 阻断项 -> 修正内容 -> 复核结果”的对应关系。
4. 二轮复核前：prompt 必须指向上一轮报告和本轮修订摘要，或直接列出上一轮结论、阻断项和修订摘要；不能只说“第一轮”。
5. 收尾时：更新状态文件和 handoff，让下一轮 agent 不依赖当前聊天记忆。

## 派单提示

```md
你是无状态 subagent。开始前读取以下共享信息库文件：
- [README / MASTER_CONTEXT / STATUS / 上一轮报告]

你的任务：[明确任务]
本轮只关注：[范围]
上一轮结论：[如果适用，写明结论，或指向包含结论的报告]
上一轮阻断项：[如果适用，逐条列出，或指向包含阻断项的报告]
输出：[报告路径或结构化结论]
```
