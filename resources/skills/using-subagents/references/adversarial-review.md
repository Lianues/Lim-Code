# 对抗性审查

当任务需要独立反驳、互相验证或多轮收敛时，读取本文件。

## 核心模型

对抗性审查不是让 Agent 互相附和，而是让它们在独立上下文中寻找失败模式。主 agent 先提出候选设计，再让不同 subagent 分别从不同角度 Block 它。

## 推荐流程

1. **写候选主张**：把设计、假设、证据和验收标准写清楚。
2. **并行反驳**：给每个 subagent 不同审查角度：官方规范、真实用户体验、代码实现、安全、测试、长期维护。
3. **限制输出**：要求每个 subagent 输出 `Approve` 或 `Block`；Block 时最多列 3-5 个阻断点，必须附证据。
4. **主 agent 修正**：只修阻断点，不因为单个偏好重写全局设计。
5. **定向复核**：把修正后的内容、上一轮阻断点、对应修正摘要和被审文件路径一起发给原审查角度复核。不要只写“第一轮阻断问题”。
6. **收敛条件**：所有关键角度 Approve，或用户明确接受剩余风险。

## Prompt 模板

```md
请独立审查以下候选设计。你不能假设自己知道本轮对话之外的上下文；所有必要背景都在本 prompt 或给定文件中。

审查角度：[官方规范 / 安全 / 测试 / 架构 / 真实用户体验]
候选设计：[粘贴或给出文件路径]
输出格式：Approve 或 Block。若 Block，列出最多 5 个阻断点，每个点包含证据、影响和建议修正。
```

## 定向复核模板

```md
你是无状态复核 subagent。不要假设自己知道“第一轮”发生了什么；以下是完整复核上下文。

被复核对象：[文件路径或设计摘要]
上一轮结论：[Approve / Block / 有条件通过 / 其他]
上一轮阻断项：
1. [阻断项 + 证据]
2. [阻断项 + 证据]
本轮修订摘要：
1. [对应修订]
2. [对应修订]
复核任务：只判断本轮修订是否解决上述阻断项，不扩展新审查范围。
输出格式：APPROVE 或 REJECT；若 REJECT，只列仍未解决的阻断项。
```

## 注意事项

- 不要让审查 Agent 直接替主 agent 做最终决策。
- 不要在第一轮审查之间共享其他 Agent 的结论；保持独立性。
- 第二轮复核可以共享阻断点和修正内容，因为目标是验证是否解决。
