# 多角色视角审查

当任务需要从多个专业视角独立思考时，读取本文件。

## 常用角色

- 架构审查者：边界、抽象、长期维护、是否有特判。
- 安全审查者：权限、secrets、外部调用、供应链、数据外泄。
- 测试审查者：回归、可复现性、失败原因、覆盖缺口。
- 性能审查者：复杂度、缓存、并发、资源占用。
- 产品/用户视角：真实体验、误用场景、文案和操作成本。
- 维护者视角：可读性、迁移路径、文档和后续排障。

## 分配规则

- 每个角色拿到同一份共享上下文，但独立输出。
- 每个角色只审自己的关注面，避免所有 Agent 都给泛泛建议。
- 输出必须包含：Approve/Block、最多 5 个问题、证据、建议修正。
- 主 agent 负责综合；不要让角色 Agent 互相投票决定最终结论。

## 汇总规则

按风险而不是票数处理结论：

- 一个高可信安全 Block 可以否决多个泛泛 Approve。
- 多个角色指出同一根因时，应合并成一个修正项。
- 对互相矛盾的建议，启动第二轮 targeted review。
